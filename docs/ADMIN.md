# Админка Turtle Cruise Cyprus — полный гайд

Админка: `https://ваш-домен/admin/`  
Интерфейс: **Django Unfold** (боковое меню слева).

Этот документ объясняет **каждый раздел**, **каждое поле** и **что с ним делать на практике**.

---

## Быстрый старт: что смотреть каждый день

| Задача | Куда идти |
|--------|-----------|
| Новые брони / оплаты | **Бронирования → Брони** (фильтр по статусу `Подтверждена`) |
| Кто не доплатил | **Брони** со статусом `Ожидает оплату` |
| Проблемы с Stripe | **Платежи**, **Webhook логи**, **API логи** |
| Письма/SMS не ушли | **Email логи** / **SMS логи** (статус `Ошибка`) |
| Закрыть дату / sold out | **Исключения дат** или `python manage.py set_cruise_capacity` |
| Поведение на сайте | **Аналитика воронки**, **Просмотры страниц** |

## 1. Бронирования

### 1.1 Брони (`Booking`)

Главная сущность — одна запись = одна попытка/факт бронирования круиза.

#### Статусы (`status`)

| Статус | Значение | Что делать |
|--------|----------|------------|
| `Черновик` | Не используется в проде обычно | Можно удалить, если мусор |
| `Ожидает оплату` | Бронь создана, клиент на шаге оплаты. **Места ещё не заняты** — только после `Подтверждена` | Ждёт оплату; при sold out закроется автоматически |
| `Подтверждена` | Оплачено, места заняты | Основной рабочий статус — сверяйте с расписанием лодки |
| `Отменена` | Отмена вручную или клиентом | Места освобождены |
| `Истекло удержание` | Hold истёк без оплаты | Места освобождены автоматически |
| `Ошибка оплаты` | Stripe вернул ошибку | Смотреть **Платежи** и **Webhook логи** |
| `Возврат` | Деньги возвращены | Бронь не активна |

#### Поля

| Поле | Что значит |
|------|------------|
| **Публичный номер брони** (`public_id`) | ID для клиента и поддержки (в письмах, SMS). Генерируется автоматически, не менять |
| **Клиент** | Ссылка на карточку клиента |
| **Круиз** | Morning / Sunset и т.д. |
| **Дата круиза** | День выезда |
| **Время начала / окончания** | Зафиксированы в момент hold из расписания |
| **Количество взрослых / детей** | Пассажиры |
| **Всего занятых мест** | adults + children (для вместимости) |
| **Цена за взрослого / ребенка** | Цена **на момент брони** (с учётом override даты) |
| **Итоговая сумма** | Сумма к оплате |
| **Валюта** | Обычно EUR |
| **Причина отмены** | `sold_out`, `hold_expired`, `user_cancelled` — для аналитики |
| **Комментарий клиента** | То, что написал в форме |
| **Заметки администратора** | Внутренние заметки, клиент не видит |
| **Источник** | `web` — с сайта, `admin` — из админки, `migration` — из WP |
| **Подтверждено / Отменено** | Временные метки перехода статуса |
| **Legacy MotoPress ID** | Связь со старым WordPress, только для миграции |

#### Типичные действия

- **Найти бронь клиента:** поиск по `public_id`, email или телефону клиента.
- **Проверить цепочку:** откройте бронь → смотрите связанные **Платежи**, **Email логи**, **Операции**.
- **Ручная отмена:** смените статус на `Отменена`, укажите причину в заметках. Осторожно: места освободятся только если логика отмены вызвана через API/сервис; прямое редактирование в админке — аварийный путь.

---

### 1.2 Клиенты (`Customer`)

| Поле | Что значит |
|------|------------|
| **Имя / Фамилия** | Из формы бронирования |
| **Email** | Для писем Brevo |
| **Телефон** | Для SMS ClickSend, формат с кодом страны |
| **Код страны телефона** | Например `+357` |
| **Согласие на маркетинг** | Opt-in на рассылки |
| **Legacy customer ID** | ID из MotoPress |

Один клиент может иметь несколько броней. Дубликаты email возможны после миграции — объединять вручную не обязательно.

---

### 1.3 Платежи (`Payment`)

Один Payment ≈ одна сессия оплаты Stripe на бронь.

#### Статусы

| Статус | Значение |
|--------|----------|
| `Нужен способ оплаты` | Intent создан, карта ещё не введена |
| `Нужно действие клиента` | 3D Secure и т.п. |
| `В обработке` | Stripe обрабатывает |
| `Успешно оплачен` | Деньги получены → бронь должна стать `Подтверждена` |
| `Ошибка оплаты` | Отказ банка / карты |
| `Отменен` | Сессия отменена |
| `Возврат` | Refund в Stripe |

#### Поля

| Поле | Что значит |
|------|------------|
| **Stripe PaymentIntent ID** | Главный ID в Stripe Dashboard |
| **Stripe Checkout Session ID** | Если оплата через Checkout |
| **Stripe Charge ID** | Конкретное списание |
| **Ключ идемпотентности** | Защита от двойного списания |
| **Stripe client secret** | Техническое, для фронта — не светить |
| **Оплачено** (`paid_at`) | Когда пришёл webhook `payment_intent.succeeded` |

**Если оплата прошла в Stripe, а бронь не подтвердилась:** смотрите **Webhook логи** (`processed=False` или `processing_error`).

## 2. Круизы

### 2.1 Круизы (`Cruise`)

Продукт: Morning Cruise, Sunset Cruise и т.д.

| Поле | Что значит | Что делать |
|------|------------|------------|
| **Код** (`code`) | `morning`, `sunset` — **не менять** после запуска, от него завязан сайт и API | |
| **Название / Описание** | Для админки и внутренних писем | |
| **Базовая цена взрослого / ребенка** | Дефолт, если нет override на дату | Меняйте здесь для глобального изменения цен |
| **Дети разрешены** | Если выкл — детский тариф недоступен | |
| **Базовая вместимость** | Макс. мест на одну дату/слот | Для теста sold out: `python manage.py set_cruise_capacity 3` |
| **Длительность, минут** | Справочно (4 ч = 240) | |
| **Активен** | Если выкл — нельзя забронировать на сайте | |
| **Порядок сортировки** | В списках админки | |

---

### 2.2 Расписания (`CruiseSchedule`)

Когда круиз **обычно** ходит.

| Поле | Что значит |
|------|------------|
| **День недели** | 0 = понедельник … 6 = воскресенье |
| **Время начала / окончания** | Слот на этот день недели |
| **Часовой пояс** | По умолчанию из настроек Django |
| **Действует с / до даты** | Сезонность (пусто = всегда) |
| **Активно** | Выкл = слот не показывается в календаре |

**Пример:** Morning cruise каждый вторник 09:00–13:00 → cruise=morning, weekday=1, start/end time.

---

### 2.3 Исключения дат (`CruiseDateOverride`)

Точечные правки на **конкретный календарный день**.

| Поле | Что делать |
|------|------------|
| **Дата закрыта** (`is_closed`) | Полный стоп продаж на этот день (погода, ремонт) |
| **Переопределение вместимости** | Например 20 вместо 30 мест |
| **Переопределение цен** | Акция / праздничный тариф |
| **Заметка** | Для себя: «шторм», «частный чартер» |

Уникальность: одна запись на пару (круиз + дата).

## 3. Аудит и аналитика

Раздел для понимания **что делают люди на сайте** и **что делает система**.

### 3.1 Аналитика воронки (дашборд)

**URL:** `/admin/audit/funnel-dashboard/`

Нормальная HTML-страница (не JSON). Период: 1 / 7 / 14 / 30 дней.

#### Карточки сверху

| Метрика | Значение |
|---------|----------|
| **Сессий** | Все посетители с cookie `tc_session` за период (даже если только смотрели галерею) |
| **Дошли до оплаты** | Сессии с `is_completed=True` |
| **Бросили бронь** | Сессии с `is_abandoned=True` (нет активности 45+ мин после шага воронки) |
| **Конверсия в оплату** | `оплатили / открыли форму` × 100% |

#### Таблица воронки

Шаги по порядку:

1. Открыл форму (`popup_open`) — нажал BOOK NOW
2. Шаг 1 — дата
3. Шаг 2 — пассажиры
4. Шаг 3 — контакты
5. Шаг 4 — оплата
6. Создал бронь (hold)
7. Начал оплату
8. Оплатил

**Почему у вас были нули при 4 сессиях:** 4 человека **ходили по сайту** (галерея, главная — это видно в «Топ страниц»), но **никто не открывал форму бронирования**. События `popup_open` не создавались → воронка пустая. Это не баг, а отсутствие действий в форме.

#### Топ страниц

Среднее время и скролл по URL — из **Просмотров страниц** (только завершённые просмотры, не «зависшие» вкладки).

---

### 3.2 Сессии посетителей (`VisitorSession`)

Один браузер ≈ одна сессия (localStorage/cookie).

| Поле | Значение |
|------|----------|
| **ID сессии** | Технический UUID с фронта |
| **Связанная бронь** | Появляется после hold |
| **IP / User-Agent / Referrer** | Откуда пришёл |
| **Последний шаг / событие** | Где застрял в форме |
| **Дошёл до подтверждения** | Успешная оплата |
| **Брошена** | Автоматически через Celery task |

**Когда смотреть:** клиент жалуется «не смог оплатить» — найдите сессию по IP или брони, откройте **События воронки** по этой сессии.

---

### 3.3 События воронки (`JourneyEvent`)

Каждое действие в форме и на сайте.

| Поле | Значение |
|------|----------|
| **Тип события** | `popup_open`, `step_view`, `booking_created`, `payment_started`, `payment_completed`, `page_exit`… |
| **Шаг** | `1`–`4` для шагов формы |
| **Код круиза** | `morning` / `sunset` |
| **Данные события** | JSON: путь страницы, ошибки, view_id и т.д. |

---

### 3.4 Просмотры страниц (`PageView`)

Аналитика «как Google Analytics lite».

| Поле | Значение |
|------|----------|
| **Путь** | `/gallery/`, `/cruise/`… |
| **Время на странице, мс** | До `page_exit` или авто-закрытия |
| **Глубина скролла, %** | 0–100 |
| **Активный просмотр** | `True` = вкладка ещё открыта |

Celery раз в ~20 мин закрывает «зависшие» просмотры.

---

### 3.5 API логи (`ApiRequestLog`)

Каждый запрос к `/api/v1/...` (кроме статики).

| Поле | Зачем |
|------|-------|
| **HTTP метод / Путь / Статус** | 4xx/5xx — искать баги |
| **Длительность, мс** | Медленные endpoint'ы |
| **Session ID / Booking public ID** | Связь с пользователем |
| **Запрос / Ответ** | Обезличенный JSON (без секретов) |
| **Ошибка** | Traceback или текст |

**Типичный кейс:** `POST /api/v1/bookings/hold/` → 400 — смотрите `response_summary` (sold out, валидация).

---

### 3.6 Операции (`OperationLog`)

Бизнес-события системы (не HTTP).

| Категория | Примеры действий |
|-----------|------------------|
| `booking` | `hold_created`, `cancelled`, `confirmed` |
| `payment` | `checkout_session_created`, `webhook_received`, `payment_failed` |
| `notification` | `email_sent`, `email_failed`, `sms_sent` |
| `page` | Служебные события страниц |

| Статус | Значение |
|--------|----------|
| `Начато` | Операция стартовала |
| `Успех` | Всё ок |
| `Ошибка` | Смотреть поле **Ошибка** и **Детали** |
| `Пропущено` | Например дубликат webhook |

**Полный трейс брони:** фильтр `booking` = нужная бронь, сортировка по времени.

## 4. Коммуникации

Все логи **только для чтения** — отправка идёт автоматически при оплате/hold.

### 4.1 Email логи (`EmailLog`)

| Поле | Значение |
|------|----------|
| **Код шаблона** | `booking_customer` — клиенту, `booking_admin` — вам |
| **Статус** | `queued` → `sent` / `failed` / `delivered` / `bounced` |
| **ID сообщения у провайдера** | Искать в Brevo |
| **Снимок payload** | Данные письма без секретов |

**Если `failed`:** поле **Ошибка** + проверьте DNS/API ключ Brevo в `.env`.

---

### 4.2 SMS логи (`SmsLog`)

Аналогично, провайдер ClickSend. Статусы: очередь / отправлено / ошибка / доставлено.

---

### 4.3 Telegram логи (`TelegramLog`)

Уведомления вам в Telegram при новой брони. **Chat ID** — куда ушло сообщение.

---

### 4.4 Webhook логи (`WebhookLog`)

Входящие события Stripe.

| Поле | Значение |
|------|----------|
| **Подпись проверена** | `False` = подозрительный запрос или неверный webhook secret |
| **Обработан** | `False` = бронь могла не обновиться |
| **Ошибка обработки** | Текст исключения |
| **Payload** | Полное тело события |

**Важно:** при `signature_valid=False` не доверяйте событию — проверьте `STRIPE_WEBHOOK_SECRET`.

## 5. Пользователи

Раздел **Users** — стандартные пользователи Django.

- **Staff** — доступ в админку
- **Superuser** — полные права

Создание: `python manage.py createsuperuser`

**Рабочие настройки сайта** (Stripe, email, SMS) — в файле **`.env`**, не в админке.  
Цены и вместимость — в **Круизы** / **Исключения дат**.

## 6. Связи между разделами (схема)

```
Посетитель сайта
    → VisitorSession + PageView + JourneyEvent
    → API: hold → Booking (Ожидает оплату)
    → Payment + Stripe Checkout
    → WebhookLog → Booking (Подтверждена)
    → EmailLog + SmsLog + TelegramLog
    → OperationLog (на каждом шаге)
```

## 7. Частые вопросы

### «Аналитика воронки показывала JSON»
Был битый пункт меню на API URL. Исправлено: теперь `/admin/audit/funnel-dashboard/`.

### «Воронка нулевая, но сессии есть»
Люди не открывали BOOK NOW. Проверьте: зайдите на `/cruise/`, нажмите кнопку бронирования — в **События воронки** должен появиться `popup_open`.

### «Мест нет, а в админке capacity 30»
Смотрите подтверждённые + pending брони на эту дату, или **Исключения дат** с `is_closed` / уменьшенной вместимостью.

### «Письмо не пришло»
1. **Email логи** по брони  
2. Статус в Brevo  
3. Спам / неверный email в **Клиенте**

## 8. Полезные команды (вне админки)

```bash
# Создать админа
python manage.py createsuperuser

# Тест sold out (вместимость 3 места)
python manage.py set_cruise_capacity 3

# Воронка в JSON (для интеграций)
curl -H "Cookie: sessionid=..." https://домен/api/v1/audit/funnel/?days=7
```

## 9. Чего НЕ делать в админке

- Не менять `code` круиза после запуска
- Не редактировать `public_id` брони
- Не копировать `provider_client_secret` клиентам
- Не ставить `Подтверждена` вручную без реальной оплаты (рассинхрон со Stripe)
- Не удалять Webhook логи при разборе инцидентов — сначала экспорт/скрин
